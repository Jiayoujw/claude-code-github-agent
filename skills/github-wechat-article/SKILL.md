# GitHub 项目微信公众号文章生成 Skill

## 用户配置

在生成文章前，确认或使用以下默认配置：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 公众号名称 | `{{公众号名称}}` | 用于文末关注引导语，用户未提供时询问 |
| 默认受众 | 对 AI 和效率工具感兴趣的普通用户 | 可选：技术开发者 / 产品经理 / 效率达人 / 学生 / 通用 |
| 封面偏好 | GitHub OpenGraph 图裁剪 900×383 | 可自定义封面来源 |

**⚠️ 全文所有 `{{公众号名称}}` 占位符在输出时必须替换为实际的公众号名称。** 用户未提供公众号名称时，在生成文章前主动询问一次。

---

## 定位与目标

本 skill 专为「GitHub 热门项目」类微信公众号内容创作设计，核心价值：
- 将技术性的 GitHub 项目转化为**面向普通读者**的中文科普内容
- 提炼项目的**实用价值**和**使用场景**，而非堆砌技术参数
- 生成符合微信平台调性的**高打开率标题**和**强互动文末 CTA**
- 输出内容直接可用于公众号发布，无需二次大改

## 输入要求

用户需提供以下任意一项或多项：
- GitHub 项目 URL（如 `https://github.com/owner/repo`）
- 项目名称
- 项目简要描述
- （可选）目标受众偏好：技术开发者 / 产品经理 / 效率达人 / 学生 / 通用
- （可选）公众号名称（如未提供，见上方用户配置）

若未提供目标受众，默认面向「对 AI 和效率工具感兴趣的普通用户」。

## 执行流程

### 第一步：项目信息收集

**若用户提供了 GitHub URL**，使用 WebFetch 抓取以下信息：
1. 项目主页（README 首屏）
2. Star 数、Fork 数、License 类型
3. 项目语言、最近更新时间
4. Release/Tag 页（了解版本成熟度）

**若只有项目名或描述**，使用 WebSearch 搜索：
- `site:github.com [项目名]` 找到官方仓库
- `[项目名] 教程 OR 使用场景 OR 案例` 找中文参考内容

**关键信息清单（必须获取）：**
- [ ] 项目用一句话能做什么
- [ ] 解决了什么痛点 / 比现有方案好在哪里
- [ ] 面向什么人群（开发者/设计师/普通用户/所有人）
- [ ] 核心功能 3-5 个
- [ ] 是否有在线体验 / Demo / 安装门槛高不高
- [ ] Star 数量级（千 / 万 / 十万+）
- [ ] 是否开源免费

### 第二步：受众价值提炼

在动笔之前，先完成以下分析（内部思考，不输出给用户）：

```
项目本质：[用 "这是一个帮你XXX的工具" 格式描述]
最大亮点：[一句话，最打动人的那个特性]
痛点对应：[用户曾经怎么解决这个问题，项目怎么让它更简单]
传播价值：[为什么粉丝会转发这篇文章？]
```

**⚠️ 不同项目绝不能用同一套视觉风格和语气。** 将项目归入以下风格类型之一。

#### 风格归入判断流程

按以下优先级判定风格：

1. **看项目本质**（最重要）：
   - 研究/学习/知识管理 → `SCHOLAR` 或 `MENTOR`
   - 数据聚合/趋势洞察 → `EXPLORER`
   - 基础工具/国产开源/性能相关 → `PRAGMATIC`
   - 框架/协议/平台 → `ARCHITECT`
   - AI 前沿/机器人/世界模型 → `FUTURIST`
   - 个人工具/本地应用 → `COMPANION`

2. **看项目背景**（微调）：
   - 作者有传奇故事 → 可偏向 `MENTOR`
   - 大厂出品定义标准 → 可偏向 `ARCHITECT`
   - 中国团队/国产之光 → `PRAGMATIC` 优先

3. **看 Star 数**（辅助判断）：
   - 10k 以下小众项目 → `COMPANION` 或 `EXPLORER` 更合适
   - 100k+ 现象级 → 可适当加大声量，但风格不变

**📖 选定风格后，读取 `references/styles.md` 获取该风格的完整规范**（配色变量、语气特征、钩子模板、h2 标题模板、CTA 模板、Emoji 规范）。

### 第三步：生成文章

按选定布局的段落顺序和风格的颜色/语气撰写，总字数控制在风格矩阵指定的范围。

**⚠️ 同时遵循第二步b（风格=配色+语气+钩子+CTA+Emoji）和第二步c（布局=结构+排列+间距+各元素格式）。两者独立组合，禁止跨风格/跨布局复用模板。**

### 第四步：内容人性化处理

完成初稿后，检查以下内容并修正：

**去 AI 味检查清单（各风格通用）：**
- [ ] 是否有"总的来说"、"值得注意的是"、"综上所述"等套话？→ 删掉（所有风格均适用）
- [ ] 段落是否过长？→ PRAGMATIC/COMPANION 控制在 60 字以内；SCHOLAR/FUTURIST 可到 80 字
- [ ] 是否有空洞形容词？→ 换成具体数据或描述。例外：FUTURIST 允许"震撼""改变未来"等词；ARCHITECT 允许"定义标准""行业标杆"等词
- [ ] 是否口语化？能用"你"的地方就用"你"，不要用"用户"（SCHOLAR 可适当例外）
- [ ] 技术术语是否有解释？读者不一定懂 API/CLI/Docker → 附上简单解释或括号说明

**语气与节奏（严格遵循第二步b选定的风格）：**
- ⚠️ 不要套用固定语气，严格使用对应风格定义的「语气特征」和「钩子模板」
- 各风格语气已预设：SCHOLAR 理性克制 / EXPLORER 好奇驱动 / PRAGMATIC 直接利落 / MENTOR 温暖真诚 / FUTURIST 震撼前瞻 / ARCHITECT 专业权威 / COMPANION 亲切热情
- 节奏由风格决定：PRAGMATIC/COMPANION 快节奏短句，SCHOLAR/FUTURIST 允许更多展开
- 数字规范：优先用阿拉伯数字，写"3个功能"而非"三个功能"

**视角按风格区分：**
- SCHOLAR："我们发现……数据表明……"（客观观察者）
- PRAGMATIC："实测下来……"（第一人称测评）
- MENTOR："我曾经也……如果你在……"（过来人视角）
- COMPANION："我发现了一个好东西，忍不住安利给你"（朋友分享）
- FUTURIST/ARCHITECT："这正在改变……"（行业观察者）
- EXPLORER："搜索背后原来……"（探索发现者）

## 输出格式

**核心原则：一份 HTML + 一套图片 + 一张对照表**

最终交付物包含：

```
📌 推荐标题（3个候选，标注推荐首选。用户选定后，标题写入 HTML）

📄 微信兼容 HTML 文件（保存为 xxx-wechat-article.html）
   - 全文零 CSS 类、零 <style>、零 <div> 做卡片
   - 所有样式内联在标签中
   - 所有图片以 Base64 Data URI 内嵌
   - 图片格式：<img src="data:image/xxx;base64,...">

🖼️ 项目图片文件（下载到 xxx-images/ 目录）
   - 封面图（需手动上传微信）
   - 项目 demo/screenshot 图（已内嵌到 HTML，目录仅作备份）

🏷️ 话题标签建议（3-5个公众号话题标签）

📊 内容质量自评
```

**HTML 输出强制要求：**
- 必须在 `<body>` 开头包含 `<h1>` 主标题：`<h1 style="font-size:24px;color:#1a1a1a;text-align:center;margin:0 0 24px;line-height:1.5;">文章标题</h1>`
- 主标题为用户选定的标题，居中，字号 24px，不可省略

## ⚠️ 微信编辑器兼容规则（极其重要）

微信公众号编辑器是**经过严格过滤的富文本编辑器**，粘贴 HTML 时大部分 CSS 会被丢弃。

### 微信会自动删除的样式

| 写法 | 结果 | 说明 |
|------|------|------|
| `<div class="xxx">` | ❌ class 全丢 | 永远不要依赖 class |
| `<div style="background:...">` | ❌ div 背景色丢掉 | div 视觉样式靠不住 |
| `<div style="border-radius:...">` | ❌ 圆角丢掉 | 同上 |
| `<style>` 标签 | ❌ 整个丢弃 | 不能放 CSS 类定义 |
| `linear-gradient(...)` | ❌ 渐变全丢 | 微信不支持渐变 |
| `display:flex` | ❌ flex 布局全丢 | 只能用 table 布局 |

### 强制转换规则

1. **禁止 `<div>` 做卡片** → 改用 `<table width="100%" cellpadding="14" cellspacing="0" border="0" style="background:#xxx;">`
2. **禁止 class** → 所有样式写为内联 `style="..."`
3. **禁止 `<style>` 标签** → 全文零 CSS 类
4. **禁止渐变、圆角、阴影** → 微信不支持
5. **禁止 flex/grid** → 用 table 替代
6. **标签徽章** → 用 `<strong style="color:#xxx;font-size:13px;">` 替代 `<span class="tag">`
7. **数据卡片** → table 两列布局，左列 label 右列 value
8. **高亮引用** → `<blockquote style="border-left:4px solid {{主题色}};background:{{数据卡底色}};">`
9. **分割线** → `<hr style="border:none;border-top:1px dashed {{主题色}};">` 替代 div

### 微信会保留的写法

✅ `<table>` + `style="background:..."` | ✅ `<blockquote style="...">` | ✅ `<h2 style="...">` | ✅ `<strong style="color:...">` | ✅ `<pre><code>` | ✅ `<img>` | ✅ `<hr>` | ✅ `<p style="text-align:center;color:...">` | ✅ `<a style="color:...">`

## 📷 图片处理规范

### Base64 内嵌方案

微信公众号**拒绝外部图片链接**但**支持 Base64 Data URI**。HTML 正文中的图片一律使用 Base64 Data URI 格式。

```html
<p style="text-align:center;margin:18px 0;">
  <img src="data:image/gif;base64,R0lGODlh..." alt="图片说明" style="max-width:100%;display:block;margin:0 auto;">
  <span style="font-size:12px;color:#999;display:block;margin-top:4px;">▲ 图片说明文字</span>
</p>
```

### 图片处理工作流

运行 `scripts/process_images.py` 处理图片（下载 → 裁剪 → 压缩 → Base64 编码）：

1. **收集图片**：从 GitHub README 提取图片链接
2. **裁剪封面**：GitHub OpenGraph 图 → 裁剪为 900×383（微信封面标准 2.35:1）
3. **压缩 GIF**：隔帧取样 + 缩放 + 调色板降色，目标每张 <5MB
4. **Base64 编码**：生成 `<img src="data:image/xxx;base64,...">` 替换 HTML 占位符
5. **封面图**：存入 `{项目名}-images/cover-github-banner.png`（封面无法通过 Base64 粘贴，需手动上传）

### 封面图优先级

1. **优先**：GitHub OpenGraph 图 `https://opengraph.githubassets.com/1/{owner}/{repo}`
2. **备选**：README 中最有代表性的 banner/logo
3. **兜底**：项目演示 GIF 的第一帧截图

## 特殊情况处理

### 项目信息不足时
1. 搜索该项目的 Issue、Discussion 了解用户反馈
2. 搜索创作者介绍/采访内容
3. 搜索同类项目对比，从差异化角度切入
4. 文章中标注"目前仍在早期阶段，功能持续完善中"

### 纯技术开发者工具
- 标题偏向"程序员专属"、"开发神器"
- 可加入代码片段（不超过 10 行）
- CTA 改为"这个工具你用过吗？你的使用场景是什么？"

### 企业级/付费项目
- 明确说明开源版功能范围
- 不做虚假宣传"完全免费"
- 可表述为"核心功能开源免费，企业版提供更多高级功能"

## 质量标准

一篇合格的文章应满足：
- 标题中有数字或具体场景，无"震惊/强烈推荐"等夸张词
- 正文中"你"的出现次数 > "用户"的出现次数
- 有至少一个真实使用场景的具体描述
- 文末有明确的互动引导（不是"欢迎关注"这种泛泛的）
- 全文没有超过 100 字的连续段落
- 对技术门槛有说明（告诉读者能不能轻松上手）
- **不同文章之间配色、钩子句式、h2 模板、CTA 模板必须有明显差异**
- **不同文章之间排版布局必须不同**：章节顺序、数据卡格式、功能展示方式、封面位置、间距密度、h2 装饰风格、分割线类型、CTA 格式等 8 个维度，每两篇至少有 4 个不同
